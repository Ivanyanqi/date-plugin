# date-plugin v1.0 第十六轮计划

## 本轮目标

把仓库从“可导出 GitHub Release 正文”继续推进到“可预演 GitHub Release 创建命令”，补齐 `release:github`，让当前版本的 tag、标题和正文可以进一步收敛为一条可执行发布命令。

## 范围

### 1. GitHub Release 创建脚本

- 新增 [scripts/release-github.mjs](../scripts/release-github.mjs)
- 默认 dry-run，只输出 `gh release create` 预览命令
- 显式传入 `--execute` 时才真正调用 GitHub CLI

### 2. 版本信息与 prerelease 语义

- 默认读取 [package.json](../package.json) 当前版本
- 当版本号包含 `-` 时自动带上 `--prerelease`
- 支持 `--version` 和 `--target` 参数

### 3. 发布流程文档同步

- 新增 `npm run release:github`
- 在 [README.md](../README.md)、[docs/release-process.md](./release-process.md) 和 [docs/release-checklist.md](./release-checklist.md) 中说明它与 `release:notes` 的关系
- 在 preflight 中纳入对 GitHub Release 脚本的存在检查

## 本轮不做

- 不自动推送 tag
- 不自动执行 `git push`
- 不在脚本中绕过用户确认直接创建远端 release

## 风险点

- `release:github --execute` 依赖本机 `gh` 登录状态
- 当前脚本默认直接读取 release notes 文件，仍需要保证正文人工润色完成
- 远端发布属于真实外部操作，因此默认必须保持 dry-run 安全行为

## 验证方式

1. 先执行缺失的 `node scripts/release-github.mjs`，对应脚本位置为 [scripts/release-github.mjs](../scripts/release-github.mjs)
2. 新增脚本后执行 `npm run release:github`
3. 验证输出的 `gh release create` 命令、tag、title 和 prerelease 标识是否正确
4. 执行 `npm run release:preflight`，确认发布链路未破坏

## 预期产出

- 当前版本可以从 release notes 一步推进到 GitHub Release 命令预演
- 发布流程更接近真正的一键式操作，但仍保留安全确认边界
