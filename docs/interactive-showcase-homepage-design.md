# date-plugin Interactive Showcase 首页设计说明

## 目标

把当前偏 demo 性质的 [index.html](../index.html) 升级为一个同时面向开发者和产品/设计评审的 interactive showcase 首页。首页不再只是“有两个 input 的测试入口”，而是一个能让访客第一时间上手体验、理解能力边界、并继续查看接入方式的产品展示页。

## 目标用户

### 开发者接入者

- 关心是否容易接入
- 关心支持哪些常用配置
- 关心当前版本是否已经足够稳定

### 产品与设计评审

- 关心组件第一眼的质感
- 关心交互是否直观
- 关心桌面与移动端体验是否成熟

## 核心设计取向

首页的第一记忆点不是品牌口号，也不是能力列表，而是“现场可玩的日期选择器”。用户进入页面后，应当先看到一个真正能点击、能切换配置、能立即感受到反馈的组件舞台，再继续阅读能力和接入说明。

## 页面结构

### 第一屏：可玩的 Hero 舞台

左侧：

- 一句清晰价值主张
- 一小段补充说明，强调桌面/移动、键盘导航、输入同步等成熟能力
- 两个主按钮：
  - 直接试玩
  - 查看接入方式
- 一组轻量能力标签，例如 `Keyboard Ready`、`Mobile Sheet`、`Theme Tokens`

右侧：

- 一个真实可交互的日期选择器试玩区
- 试玩区包括：
  - 主输入框
  - 可切换的展示配置
  - 当前模式说明
  - 变化回显区，例如当前 locale / format / 选择结果

第一屏的主要任务是让用户“马上动手”，而不是先阅读。

### 第二屏：能力剧场

这一区域不做冗长表格，而是做卡片化说明：

- 输入同步
- 键盘导航
- 日期约束
- Today / Clear
- 移动端 sheet
- 主题变量

每张卡片既要有一句人话描述，也要有小而明确的状态词，让产品和开发者都能快速扫读。

### 第三屏：开发者快速接入

这里提供：

- 最小接入代码示例
- 模块化 API 与 legacy 入口说明
- 常用 options 提示
- 测试和发布脚本入口

目标是让开发者在试玩之后，可以立刻知道“怎么接入、怎么继续看文档”。

### 第四屏：文档入口与版本状态

展示：

- README / 发布流程 / 兼容性 / release notes 等链接提示
- 当前 beta 状态说明
- 当前验证方式摘要

这一区域承担“收尾”和“继续深挖”的职责。

## 视觉方向

首页采用“编辑感 + 工具感”的混合气质：

- 背景以浅暖白和冷灰蓝为主，避免传统 demo 页的工具感过重
- 用更有杂志感的标题排版，配合克制的技术标签
- 内容区块采用大留白、清晰栅格和少量强调色块
- 组件舞台本身是视觉中心，页面装饰不能喧宾夺主

## 交互设计

### Hero 试玩区

试玩区提供几组低成本、高感知的切换能力：

- locale 切换
- format 切换
- `showToday` / `showClear`
- `closeOnSelect`

切换后应当立即重建或刷新实例，让用户真实感受到差异。

### 页面导航

按钮应支持滚动到指定区块：

- `立即试玩` 滚动到试玩区
- `快速接入` 滚动到接入区

### 状态反馈

首页应展示当前选择值、配置摘要和关键行为说明，避免用户操作后没有反馈。

## 实现边界

这轮只改 showcase 首页层，不改日期选择器核心行为定义：

- 可以重写 [index.html](../index.html)
- 可以增加首页专属样式
- 可以用现有 [date.plugin.ext.js](../date.plugin.ext.js) 做真实交互
- 不修改 [regression.html](../regression.html) 的职责
- 不把发布测试页和首页展示页混在一起

## 验收标准

- 首页第一屏可以直接操作真实日期选择器
- 页面同时对开发者和产品/设计评审友好
- 首页信息结构明显强于当前 demo 页
- Showcase 升级不影响现有浏览器回归链
