# date-plugin v1.0 第十二轮计划

## 本轮目标

把仓库从“具备发布骨架”继续推进到“具备最小工程化发布能力”，补齐正式发布流程文档、浏览器自动回归脚本和 GitHub Actions 工作流，让现有测试页可以被重复执行和持续验证。

## 范围

### 1. 发布流程文档

- 新增 `docs/release-process.md`
- 明确版本推进、文档更新、回归执行、发布前后检查步骤
- 把“手工回归”与“CI 自动回归”串成一条统一流程

### 2. 自动回归骨架

- 新增浏览器回归脚本
- 统一串联 `tests/core`、`tests/state`、`tests/interaction`、`tests/accessibility`、`tests/public`、`tests/options` 与 `regression.html`
- 约定以页面 `#summary` 的最终结果作为通过标准

### 3. GitHub Actions

- 新增 `.github/workflows/browser-regression.yml`
- 在 push / pull request 时启动静态服务并执行浏览器回归
- 为后续正式版发布提供最小可用的仓库自验能力

### 4. 文档与元数据同步

- 更新 `package.json` 的仓库元数据与 CI 脚本
- 更新 `README.md` 的验证与发布说明
- 更新发布检查清单和包清单测试页

## 本轮不做

- 不接入 npm 自动发布
- 不新增构建产物打包目录
- 不引入 TypeScript、bundler 或测试框架迁移
- 不把现有 HTML 测试页重写为单独的单元测试体系

## 风险点

- 当前本地 `node` 运行时仍不可用，因此这轮自动化主要面向 CI 环境设计
- 浏览器测试页依赖异步 UI 与动画状态，自动脚本必须等待 `#summary` 稳定收敛
- `regression.html` 与 options 测试页使用 legacy + module 混合形态，CI 需要通过静态服务统一访问

## 验证方式

1. 静态检查新增脚本、工作流和文档引用是否闭环
2. 回归 `tests/public/package-manifest.test.html`，确认新增发布文档入口可见
3. 保持现有 `regression.html` 与各测试页列表不变，供浏览器与 CI 共用
4. 最终以 GitHub Actions 工作流作为后续发布前的标准回归入口

## 预期产出

- 仓库具备最小可执行的浏览器自动回归链路
- 发布流程从“文档分散”收敛为“一份流程文档 + 一份检查清单”
- 后续可以继续进入 `v1.0.0-beta.2` 或正式版前的发布准备阶段
