# date-plugin v1.0 第十三轮计划

## 本轮目标

把仓库从“自动回归已接通”继续推进到“发布前自检可本机执行”，补齐 CLI 版 release preflight，让版本、文档、入口、锁文件和工作流依赖不再只靠人工记忆检查。

## 范围

### 1. 发布前自检脚本

- 新增 `scripts/release-preflight-check.mjs`
- 校验 `package.json` 元数据、导出入口、发布脚本与发布文档
- 校验 `package-lock.json`、`.gitignore` 和关键工作流脚本存在

### 2. npm 脚本与 CI 串联

- 新增 `npm run release:preflight`
- 在 GitHub Actions 中先执行 preflight，再执行浏览器自动回归
- 依托 `package-lock.json` 切换为 `npm ci`

### 3. 当前文档状态收口

- 清理 README 与发布文档中“本地 node 仍不可用”的过期表述
- 明确“本机 CLI 自检 + 浏览器回归 + CI 回归”的分层验证路径

## 本轮不做

- 不接入自动版本号递增
- 不接入自动打 tag 或自动生成 GitHub Release
- 不新增 bundler 或产物打包目录

## 风险点

- CLI preflight 只能覆盖元数据与文件闭环，不能替代真实 UI 回归
- 工作流切到 `npm ci` 后，必须保证 `package-lock.json` 持续与 `package.json` 一致
- 发布文档如果继续快速迭代，容易再次出现“脚本已存在但文档仍写未完成”的漂移

## 验证方式

1. 先执行不存在的 `node scripts/release-preflight-check.mjs`，确认当前能力确实缺失
2. 新增脚本后执行 `npm run release:preflight`
3. 执行 `npm run test:ci`
4. 静态检查 README、发布流程与发布清单的命令入口是否一致

## 预期产出

- 仓库具备稳定可执行的发布前自检入口
- CI 在真正跑浏览器回归前先拦截掉元数据和文档缺失问题
- v1.0 后续发布准备从“人工检查列表”进一步收敛为“脚本 + 文档 + CI”三层闭环
