# date-plugin v1.0 第十一轮计划

## 本轮目标

把仓库从“功能已完成”推进到“具备发布骨架”，补齐 [package.json](../package.json)、导出策略和发布检查文档，让 v1.0 开始具备真实对外交付结构。

## 范围

### 1. 包元数据

- 新增 [package.json](../package.json)
- 定义 `name`、`version`、`type`、`exports`
- 增加最小 `files` 白名单和脚本说明

### 2. 导出策略

- 模块入口导出 `DatePicker`
- 样式入口导出 [date.style.css](../date.style.css)
- legacy 手动初始化入口导出 [date.plugin.ext.js](../date.plugin.ext.js)
- auto 初始化入口导出 [date.plugin.js](../date.plugin.js)

### 3. 发布文档

- 新增 [docs/release-checklist.md](./release-checklist.md)
- 明确兼容性、QA、包结构和发布动作
- [README.md](../README.md) 同步补充发布入口和相关测试页

## 本轮不做

- 不接入 npm 发布命令自动化
- 不生成构建产物目录
- 不补充许可证文件
- 不补充 CI 工作流
- 不做 CDN 分发脚本

## 风险点

- 浏览器模块缓存会影响新增 API 测试，需要让测试入口显式拉取最新实现
- `exports` 设计要兼顾 ESM 与当前 legacy 文件结构
- 发布文档承诺必须和真实入口一致，不能写超前能力

## 验证方式

1. 新增 package manifest 测试页
2. 回归 public API 与 month control 测试
3. 回归现有 options / mobile / regression 测试
4. 手动检查 [README.md](../README.md) 与 [package.json](../package.json) 导出的一致性

## 预期产出

- 仓库具备基本可发布骨架
- 外部接入者能清晰看到模块、样式和 legacy 入口
- 后续可以继续进入最终发布清单、许可证和 CI 收尾
